# AGENT.md

## 必须优先执行的交付规则

本项目的任何改动都默认要部署到生产服务器。这里的“任何改动”包括代码、样式、文案、文档、配置示例、脚本和本文件本身。

除非用户明确说“只做本地改动”“不要部署”或“先别发到服务器”，否则不能只完成本地修改、不能只启动 localhost、不能只口头说明已经改好。最终回复用户前必须尝试部署到生产服务器，并报告部署和线上检查结果。

标准交付流程：

1. 修改本地文件。
2. 运行 `npm run check`。
3. 运行 `powershell -ExecutionPolicy Bypass -File scripts/deploy.ps1`。
4. 运行 `powershell -ExecutionPolicy Bypass -File scripts/server-status.ps1`。
5. 最终回复中明确说明：本地检查是否通过、部署是否成功、线上健康检查是否通过。

如果部署或线上检查失败，不要把任务说成“已完成”。最终回复必须写清楚失败命令、失败原因、哪些内容已经留在本地、线上是否尚未更新。

部署信息以 `SERVER_HANDOFF.md` 为准。不要把 `.env`、SSH 私钥、生产数据、API key 或 admin token 写入 git、文档或聊天内容。

## 项目定位

AI费曼教室是一个面向理工科学生的学习工作台。用户把一个知识点或题目讲给“AI 学生”听，AI 学生通过追问暴露概念漏洞、条件缺失、逻辑跳跃、公式误用和表达含混。

产品不是让 AI 替用户讲课，也不是生成标准答案。核心体验必须始终围绕“用户讲给 AI 学生听，然后被追问，再生成诊断报告”。

更完整的产品说明见 `PRODUCT.md`，部署和服务器细节见 `SERVER_HANDOFF.md`。

## 当前核心闭环

必须保持这个流程：

1. 新对话：用户输入学科，选择“知识点讲解”或“题目讲解”，并填写主题。
2. 输入辅助：用户可以输入文本、公式，也可以直接上传或粘贴图片/PDF 学习材料。文字型 PDF 走文本理解，扫描版/图片型 PDF 由后端自动渲染页面并用视觉模型理解；AI 应理解材料并自动创建或补充 AI 学生对话，不要求用户先编辑识别文本或转换 PDF。
3. 讲解阶段：AI 学生先请用户用自己的话讲；后续每轮只围绕最关键的 1-3 个问题追问。
4. 报告阶段：AI 总结本次讲解暴露出的理解漏洞、已讲清楚部分和下一轮重讲任务。
5. 对话记录：会话和报告保存到左侧记录，后续可继续进入。

## AI 学生行为边界

- AI 学生是“认真、有基础、但当前知识点不会的同学”，不是老师、裁判或答案生成器。
- 不评分、不排名、不分等级，不输出通过/不通过判断。
- 不直接给完整证明、完整解题过程或标准答案。
- 要先判断用户最新输入是否修正了旧问题，再提出新的追问。
- 如果用户只给公式，追问变量含义、适用条件、来源或直观含义。
- 如果用户只给结论，追问中间推理为什么成立。
- 如果用户说法有误，用学生口吻提出疑问，不要变成批改作业。
- 每轮优先追问一个主问题，最多保留 1-3 个观察点。

## 技术约束

- 后端是原生 Node HTTP 服务，入口为 `server.js`。
- 前端主要文件是 `index.html`、`app.js`、`styles.css`。
- 依赖变更必须同步更新 `package.json` 和 `package-lock.json`。
- AI 接口由后端代理调用，前端不能暴露 API key。
- 模型输出尽量保持结构化 JSON，便于前端稳定渲染和追踪观察点。
- 图片、PDF 等上传内容统一作为学习材料处理，后端生成 `materialContext` 供 AI 学生追问使用；扫描版/图片型 PDF 必须在网页内直接处理，不要把用户流程退回到“先编辑 OCR 文本”或“先转换成文字 PDF”。

## 最终报告要求

最终报告只做学习诊断，不包含分数、等级、排名或考试化评价。报告必须具体对应本次对话，不写空泛学习建议。

报告应包含：

- 本次讲解主题。
- 主要理解漏洞。
- 关键逻辑跳跃。
- 已经讲清楚的部分。
- 仍需复习的问题。
- 下一轮重讲任务。

如果某个漏洞在对话后半段已经被用户修正，应放入“已经讲清楚的部分”，不要继续当作未解决漏洞。
